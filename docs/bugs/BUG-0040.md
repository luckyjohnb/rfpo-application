## 2. Root Cause Analysis

The bug is a **missing fallback/error handling** in `advance_to_next_step()` in `models.py` (line ~2034).

The method has three "end of workflow" code paths — all in the `else` branch when `current_stage_order >= len(stages)`:

1. **Parallel stage, end of stages** (line ~2073): calls `check_completion_status()`, only acts if result is truthy
2. **Sequential stage, end of stages** (line ~2092): calls `check_completion_status()`, only acts if result is truthy
3. **Top-of-method early return** (line ~2037-2046): calls `check_completion_status()` and returns early for "approved" or "refused"

The problem occurs in paths 1 and 2: when `check_completion_status()` returns `None`, the code does **nothing** — no status change, no error, no logging. The workflow is left in a state where:
- `overall_status` = "waiting"
- `completed_at` = None
- All stage pointers are exhausted (no more stages to advance to)
- `check_completion_status()` keeps returning `None` because orphaned pending actions exist

**Why does `check_completion_status()` return `None`?**
It returns `None` when there are still `pending` actions (line ~2017: `if pending_actions: return None`) OR when there are no completed actions at all (line ~2021: `if not completed_actions: return None`). 

The typical scenario: a workflow stage was modified after action records were created, leaving orphaned `pending` actions that no longer correspond to any stage/step in the workflow structure. All "real" actions are completed, but these orphans keep `check_completion_status()` returning `None`.

**This is a defensive programming gap.** The method should handle the "end of stages but still not complete" case with either:
- A forced completion (treat as approved if all non-orphaned actions are done)
- An error status (mark as "error" or "stuck" so admins can see it)
- Logging a warning for admin intervention

## 4. Fix Strategy

### Phase 1: Core Fix — Handle the "end of stages + incomplete" case

When `advance_to_next_step()` reaches the end of all stages but `check_completion_status()` returns `None`, the method should detect and resolve the stuck state rather than silently doing nothing.

**Approach:** After the pointer reaches the end of stages and `check_completion_status()` returns `None`, check if all actions matching actual workflow stages are complete. If so, force-complete any orphaned pending actions and mark the workflow as approved.

```python
# At end of both parallel and sequential branches:
else:
    # Reached end of workflow structure
    final_status = self.check_completion_status()
    if final_status:
        self.overall_status = final_status
        self.completed_at = datetime.utcnow()
    else:
        # LIMBO GUARD: stages exhausted but check_completion_status
        # returned None — likely orphaned pending actions.
        # Force-resolve orphaned actions and re-check.
        self._resolve_orphaned_actions()
        retry_status = self.check_completion_status()
        if retry_status:
            self.overall_status = retry_status
            self.completed_at = datetime.utcnow()
        else:
            # Truly stuck — mark as error for admin review
            import logging
            logging.getLogger(__name__).error(
                "Workflow %s stuck at end of stages — marking as error",
                self.instance_id,
            )
            self.overall_status = "error"
            self.completed_at = datetime.utcnow()
```

### Phase 2: Add `_resolve_orphaned_actions()` helper

```python
def _resolve_orphaned_actions(self):
    """Auto-complete pending actions that don't match any stage/step
    in the workflow structure (orphans from stage edits)."""
    data = self.get_instance_data()
    stages = data.get("stages", [])
    valid_positions = set()
    for stage in stages:
        stage_order = stage.get("stage_order")
        for step in stage.get("steps", []):
            valid_positions.add((stage_order, step.get("step_order")))

    for action in self.actions:
        if action.status == "pending":
            pos = (action.stage_order, action.step_order)
            if pos not in valid_positions:
                action.status = "approved"
                action.completed_at = datetime.utcnow()
                action.comments = "Auto-resolved: orphaned action (no matching stage/step)"
```

### Phase 3: Hardening — Add the same guard to the top-of-method early return

The early return at the top of `advance_to_next_step()` (line ~2037-2046) already checks for "refused" and "approved". Add an `else` branch there too for the `None` case when stages are exhausted, as a secondary safety net.

### Phase 4: Test coverage
Add tests for the limbo scenario in `tests/integration/test_api_approvals.py`.

**Edge case tests:**

## 8. Regression Prevention

- The orphaned action resolver should be idempotent — calling it multiple times has no additional effect
- Add logging when orphans are detected so ops can investigate the root cause
- Consider adding a health check endpoint that detects workflows stuck in "waiting" with exhausted stage pointers
- The "error" status should be visible in the admin panel so admins know to investigate

## 9. Security Considerations

- The auto-resolution of orphaned actions is a privilege escalation concern if malicious actions could be injected as "orphaned" — but action creation is admin-only, so this is not a realistic attack vector
- The "error" status should not be exposed to non-admin users; verify the RFPO explorer filters appropriately
- No input validation changes needed — this is internal model logic
