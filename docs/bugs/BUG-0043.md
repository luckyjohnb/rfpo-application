## 2. Root Cause Analysis

The root cause is a **lack of centralized status constants** across three distinct status domains:

1. **RFPO statuses** (title-case): `"Draft"`, `"Submitted"`, `"Pending Approval"`, `"Approved"`, `"Refused"`, `"Cancelled"` — used in `simple_api.py`, `custom_admin.py`, and `models.py` (`RFPO.status` column default `"Draft"`).

2. **Instance statuses** (lowercase): `"draft"`, `"waiting"`, `"approved"`, `"refused"`, `"cancelled"`, `"withdrawn"` — used in `models.py` (`RFPOApprovalInstance.overall_status`), `simple_api.py`, and `custom_admin.py`.

3. **Action statuses** (lowercase): `"pending"`, `"approved"`, `"conditional"`, `"refused"`, `"cancelled"` — used in `models.py` (`RFPOApprovalAction.status`), `simple_api.py`, `custom_admin.py`, and `api/report_routes.py`.

There is a `List` model (RFPO_STATU) in `sqlalchemy_db_init.py` that defines reference statuses, but it is **never used for validation** — it's a UI dropdown data source only.

All status assignments are bare string literals scattered across 4+ files with 50+ occurrences. The inconsistent casing (title-case for RFPO, lowercase for Instance/Action) makes this especially error-prone since functions switching between entity types can easily apply the wrong case.

## 3. Impact Assessment

**Affected files/functions:**
- `models.py` — RFPO model default (`"Draft"`), Instance model default (`"draft"`), Action model default (`"pending"`), `advance_to_next_step()`, `check_completion_status()`
- `simple_api.py` — `take_approval_action()`, `bulk_approval_action()`, `submit_rfpo()`, `withdraw_rfpo()`, status comparisons in queries
- `custom_admin.py` — approval action handlers, RFPO submission, status filters, dashboard stats queries
- `api/report_routes.py` — report queries filtering by action status
- `api/admin_routes.py` — pending action queries

**User-facing impact:** Currently minimal — no known typos exist in production code. This is a **latent defect** that increases the risk of future regressions.

**Data impact:** If a typo were introduced, the affected record would have an invalid status string that would be invisible to all filters and queries, effectively creating a "ghost" record.

**Scope:** Every RFPO, every approval instance, every approval action — the entire approval workflow depends on correct status strings.

## 4. Fix Strategy

### Phase 1: Create Status Constants Module (Core Fix)
Create a `status_constants.py` module defining frozen dataclasses or simple class-based constants for all three status domains. This provides a single source of truth and IDE autocompletion without requiring database schema changes.

```python
# status_constants.py

class RFPOStatus:
    """RFPO-level statuses (title-case, matching existing data)."""
    DRAFT = "Draft"
    SUBMITTED = "Submitted"
    PENDING_APPROVAL = "Pending Approval"
    APPROVED = "Approved"
    REFUSED = "Refused"
    CANCELLED = "Cancelled"

    ALL = frozenset({DRAFT, SUBMITTED, PENDING_APPROVAL, APPROVED, REFUSED, CANCELLED})

class InstanceStatus:
    """Approval instance statuses (lowercase)."""
    DRAFT = "draft"
    WAITING = "waiting"
    APPROVED = "approved"
    REFUSED = "refused"
    CANCELLED = "cancelled"
    WITHDRAWN = "withdrawn"

    ALL = frozenset({DRAFT, WAITING, APPROVED, REFUSED, CANCELLED, WITHDRAWN})

class ActionStatus:
    """Approval action statuses (lowercase)."""
    PENDING = "pending"
    APPROVED = "approved"
    CONDITIONAL = "conditional"
    REFUSED = "refused"
    CANCELLED = "cancelled"

    ALL = frozenset({PENDING, APPROVED, CONDITIONAL, REFUSED, CANCELLED})
```

### Phase 2: Replace String Literals with Constants
Replace all ~50 bare string literal status references across the codebase with the corresponding constant. This is a large-but-mechanical find-and-replace operation.

### Phase 3: Add Validation (Hardening)
Add `@validates` decorators on the three status columns in `models.py` to reject invalid values at the ORM level:

```python
from sqlalchemy.orm import validates

class RFPO(db.Model):
    @validates('status')
    def validate_status(self, key, value):
        if value not in RFPOStatus.ALL:
            raise ValueError(f"Invalid RFPO status: {value!r}")
        return value
```

This catches typos at write-time with a clear error instead of silent corruption.

## 5. Database Impact

No database changes required. Status column types and existing data values remain unchanged. The constants merely document and enforce the existing string values.

## 8. Regression Prevention

- The `@validates` decorators will catch future typos at write-time with a clear `ValueError`
- IDE autocompletion on constants (e.g., `RFPOStatus.APPROVED`) prevents typos at development time
- Add a lint/grep check to CI that flags bare status string literals in `.py` files (optional)
- The `ALL` frozensets serve as documentation of valid values

## 9. Security Considerations

No security impact. Status values are internal system state, not user-facing input. The validation decorators add defense-in-depth but the status fields are never directly user-controlled (they're set by server-side logic only).
