## 2. Root Cause Analysis

The `_get_authorized_approver_ids()` function at `simple_api.py` line 145-169 has a **bare `except Exception: pass`** block that catches ALL exceptions without logging or alerting:

```python
def _get_authorized_approver_ids(instance, action):
    authorized = {action.approver_id}
    try:
        data = instance.get_instance_data()
        for stage in data.get("stages", []):
            if stage.get("stage_order") == action.stage_order:
                for step in stage.get("steps", []):
                    if step.get("step_order") == action.step_order:
                        backup_id = step.get("backup_approver_id")
                        if backup_id:
                            authorized.add(backup_id)
                        break
                break
    except Exception:
        pass  # <-- THE BUG: silent swallow
    return authorized
```

**Why this is a defect:**

1. **Silent failure:** If `instance.get_instance_data()` returns malformed data or the JSON is corrupt, the backup approver is excluded from the authorized set with absolutely no logging. The model's own `get_instance_data()` already handles `json.JSONDecodeError` by returning `{}`, but if the JSON is valid but structurally wrong (e.g., `stages` is a string instead of a list), the `.get()` / iteration throws `TypeError` or `AttributeError` — and that gets silently swallowed.

2. **Invisible 403:** The backup approver receives a `403 Forbidden` response. The security log at line 1258 logs "User X attempted to approve action Y (authorized: {only_primary})" — but doesn't explain WHY the backup was excluded. Admins see the security warning but have no way to know it was caused by data corruption vs. a legitimately unauthorized access attempt.

3. **Two call sites affected:** The function is called at line 1252 (single approval action) and line 1565 (bulk approval), so both individual and batch approvals are affected.

4. **Root type: Error suppression anti-pattern.** This is a bare `except: pass` that violates Python best practices (PEP 8: "Never use bare except"). The intent was safe fallback — only primary approver authorized if data is bad — but the implementation loses all diagnostic information.

## 3. Impact Assessment

- **Affected files/functions:**
  - `simple_api.py`: `_get_authorized_approver_ids()` (line 145)
  - Called from `approval_action()` (line 1252) and bulk approval (line 1565)
- **User-facing impact:** Backup approver gets 403 Forbidden with no explanation. They cannot approve the RFPO even though they are configured as backup. This blocks the entire approval workflow for that RFPO.
- **Data impact:** No data corruption from this bug — it's a read-path issue. However, the underlying JSON corruption that triggers it may indicate a separate write-path problem.
- **Scope:** Any RFPO with a corrupted or malformed `instance_data` snapshot where a backup approver tries to act. Low frequency under normal operation, but catastrophic when it occurs because the workflow is blocked with no diagnostic trail.

## 4. Fix Strategy

### Phase 1: Core fix — Replace bare except with proper logging

Replace `except Exception: pass` with structured error logging that captures:
- The exception type and message
- The instance_id and action_id for traceability
- The stage_order and step_order being searched

The function should STILL return `{action.approver_id}` on failure (safe fallback to primary-only authorization) but must LOG the error so admins can diagnose the problem.

```python
def _get_authorized_approver_ids(instance, action):
    authorized = {action.approver_id}
    try:
        data = instance.get_instance_data()
        for stage in data.get("stages", []):
            if stage.get("stage_order") == action.stage_order:
                for step in stage.get("steps", []):
                    if step.get("step_order") == action.step_order:
                        backup_id = step.get("backup_approver_id")
                        if backup_id:
                            authorized.add(backup_id)
                        break
                break
    except Exception as e:
        app.logger.error(
            "Failed to resolve backup approver from instance_data — "
            "instance_id=%s, action_id=%s, stage_order=%s, step_order=%s: %s",
            getattr(instance, 'instance_id', '?'),
            getattr(action, 'action_id', '?'),
            getattr(action, 'stage_order', '?'),
            getattr(action, 'step_order', '?'),
            str(e),
        )
    return authorized
```

**Key decision:** Keep the safe fallback (primary-only) rather than raising. An authorization check must never crash the request — but it MUST log the failure.

### Phase 2: Hardening — Add data structure validation

Add lightweight validation of the `instance_data` structure before iterating:

```python
if not isinstance(data, dict) or not isinstance(data.get("stages"), list):
    app.logger.warning(
        "Malformed instance_data for instance_id=%s: expected dict with 'stages' list, got %s",
        getattr(instance, 'instance_id', '?'),
        type(data).__name__,
    )
    return authorized
```

This catches the most common structural corruption scenario without needing the broad except.

## 9. Security Considerations

- **Authorization fallback is correct:** On error, the function falls back to primary-only authorization (more restrictive, not less). This is secure.
- **No information leakage:** The error details go to server logs only. The 403 response to the user remains generic.
- **Audit trail improvement:** The fix adds logging that makes security investigations possible when backup approvers report access issues.
