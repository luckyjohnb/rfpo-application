## 2. Root Cause Analysis

The reminder/escalation **application code is fully implemented** — `reminder_service.py` contains working `run_reminders()` and `run_escalations()` functions that query overdue `RFPOApprovalAction` records, send reminder emails (up to `MAX_REMINDERS_BEFORE_ESCALATION`), then auto-escalate to backup approvers. The `Dockerfile.reminder` exists and the Azure Bicep infrastructure (`azure/main.bicep`) defines a `rfpo-reminder-job` Container Apps Job with a daily cron schedule (`0 8 * * *`).

However, three deployment gaps prevent it from ever running:

1. **Deployment script gap**: `redeploy-phase1.sh` builds and pushes the `rfpo-reminder` image to ACR (line 131), but only updates the three Container Apps (`rfpo-api`, `rfpo-admin`, `rfpo-user`) — it never calls `az containerapp job update` to update the reminder job with the new image. The job continues running with whatever stale image was last deployed (or never runs if the Bicep template was never deployed after the job resource was added).

2. **Missing email environment variables**: The Bicep template passes only `DATABASE_URL`, `CRON_TZ`, `REMINDER_REPEAT_DAYS`, and `MAX_REMINDERS_BEFORE_ESCALATION` to the reminder job container. The email service requires `ACS_CONNECTION_STRING` + `ACS_SENDER_EMAIL` (or `MAIL_SERVER`/`MAIL_USERNAME`/`MAIL_PASSWORD`) to send emails. Without these, `send_approval_reminder()` and `send_escalation_notification()` will fail silently or throw.

3. **No job provisioning verification**: There is no health check or deployment verification step that confirms the Container Apps Job was actually created and is running on schedule. The `redeploy-phase1.sh` health check loop only checks the three web apps.

Additionally, the admin panel has a manual trigger endpoint (`POST /trigger-reminders`) at `custom_admin.py:2181` that imports and calls `run_reminders()` and `run_escalations()` directly — this works because it runs in the admin app's context which has all env vars. But this is a manual workaround, not a scheduled solution.

## 3. Impact Assessment

**Affected files/functions:**
- `redeploy-phase1.sh` — missing job update step
- `azure/main.bicep` — missing email service env vars on reminder job
- `reminder_service.py` — code is correct but never executes in production
- `custom_admin.py:2181` — manual trigger works but is not automated

**User-facing impact:** An approver who ignores their approval task blocks the entire RFPO workflow indefinitely. No reminders are sent. No escalation to backup approvers occurs. Submitters see their RFPO stuck in "Pending Approval" with no recourse except manually contacting the approver or an admin.

**Data impact:** No data corruption. The `reminder_count`, `last_reminder_sent_utc`, `is_escalated`, `escalated_at`, and `escalation_reason` fields remain at their defaults (0, null, false, null, null) for all actions.

**Scope:** Every RFPO approval workflow is affected. Any approval step with `timeout_days` and `auto_escalate=True` is a dead configuration.

## 4. Fix Strategy

### Phase 1: Wire the reminder job into the deployment pipeline (Core fix)
Add an `az containerapp job update` step to `redeploy-phase1.sh` after the three Container App updates. This ensures the reminder job image is updated on every deployment.

```bash
# After updating the three Container Apps, update the reminder job
echo "Updating rfpo-reminder-job..."
az containerapp job update \
    --name "rfpo-reminder-job" \
    --resource-group "$RESOURCE_GROUP" \
    --image "$ACR_LOGIN_SERVER/rfpo-reminder@${rfpo_reminder_DIGEST}" \
    --output none
```

### Phase 2: Add email service env vars to the Bicep template (Hardening)
Add the required email service secrets and environment variables to the `reminderJob` resource in `azure/main.bicep`:

```bicep
secrets: [
  { name: 'acr-password', value: acr.listCredentials().passwords[0].value }
  { name: 'database-url', value: '...' }
  { name: 'acs-connection-string', value: acsConnectionString }  // NEW
  { name: 'acs-sender-email', value: acsSenderEmail }            // NEW
]
// In container env:
{ name: 'ACS_CONNECTION_STRING', secretRef: 'acs-connection-string' }
{ name: 'ACS_SENDER_EMAIL', secretRef: 'acs-sender-email' }
```

Alternatively, pass these via `az containerapp job update --set-env-vars` in the redeploy script, matching the pattern used for the other Container Apps.

### Phase 3: Add deployment verification and observability
- Add the reminder job to the health check loop in `redeploy-phase1.sh` (verify last execution status via `az containerapp job execution list`)
- Add a `/api/admin/reminder-status` endpoint that returns the last execution time and outcome (queryable from the admin dashboard)
- Log a warning on admin dashboard when the reminder job hasn't run in >48 hours

## 5. Database Impact

No database changes required. All necessary columns already exist on `RFPOApprovalAction` (`reminder_count`, `last_reminder_sent_utc`, `is_escalated`, `escalated_at`, `escalation_reason`) and `RFPOApprovalStep` (`timeout_days`, `auto_escalate`).

## 7. Testing Strategy

**Regression tests:**
- Unit test `run_reminders()` with mocked DB and email service: verify it queries overdue actions, sends reminders, increments `reminder_count`, and updates `last_reminder_sent_utc`
- Unit test `run_escalations()`: verify it escalates actions where `reminder_count >= MAX_REMINDERS` and calls `escalate_action()` with correct reason
- Integration test: create an overdue pending action, run `run_reminders()`, verify email sent and fields updated

**Edge case tests:**
- Action already escalated → should be skipped
- Action with no backup approver → escalation marks as escalated but doesn't send backup email
- Approver has no email address → graceful skip with warning log
- Instance not in "waiting" status → actions should be skipped
- `reminder_count` already at max → should trigger escalation instead of reminder

**Manual verification:**

## 9. Security Considerations

- **Secrets management**: ACS connection string must be stored as a Bicep secret (secretRef), never as plain-text env vars. The current Bicep pattern for `database-url` already uses secrets correctly — follow the same pattern.
- **Least privilege**: The reminder job only needs DB read/write for `RFPOApprovalAction` and `Notification` tables, plus email sending capability. No admin or API access needed.

| Risk | Severity | Mitigation |

| Reminder job sends duplicate emails if cron runs twice | Medium | `REMINDER_REPEAT_DAYS` interval check + `last_reminder_sent_utc` timestamp prevents double-sending within the repeat window |
