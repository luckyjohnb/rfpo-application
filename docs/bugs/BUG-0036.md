## 1. Bug Details

| Field | Value |

| **Title** | Reassignment does not reset due_date for new approver |
| **Description** | When an approval action is reassigned to a new approver via `/api/users/reassign-approval/<action_id>`, the code updates `assigned_at` but never recalculates `due_date`. The new approver inherits the original approver's deadline, which may have already expired. This means a freshly reassigned action could immediately appear as overdue. |

## 2. Root Cause Analysis

The `reassign_approval_action()` function at `simple_api.py` line 2131 updates `assigned_at` but never recalculates `due_date`:

```python
# Reassign (line ~2172-2175)
action.approver_id = new_approver.record_id
action.approver_name = new_approver.get_display_name()
action.assigned_at = datetime.utcnow()     # ← Updated
action.updated_at = datetime.utcnow()      # ← Updated
# action.due_date ← NEVER RECALCULATED     # ← THE BUG
```

**How `due_date` is initially set** (line 1928):
```python
due_date=datetime.utcnow() + timedelta(days=step_data.get("timeout_days", 5)),
```

The `timeout_days` value is stored in the instance_data snapshot at the step level (line 1800):
```python
"timeout_days": step.timeout_days,
```

**Why this is a defect:**

1. **Stale deadline:** When an action is reassigned, the new approver gets the original approver's deadline. If the old approver held the action for 4 out of 5 days, the new approver gets only 1 day — despite having just received it.

2. **Immediately overdue:** If the action is reassigned after the timeout period (e.g., reassigned on day 6 with a 5-day timeout), the new approver receives an action that is ALREADY overdue. The reminder system (`Dockerfile.reminder`) will immediately flag it as overdue and may send escalation notices.

3. **Stale reminder state:** The function also fails to reset `last_reminder_sent_utc`, `reminder_count`, `is_escalated`, and `escalated_at`. If the previous approver had already been reminded 3 times and escalated, the new approver inherits that state — they may never receive a first reminder because `reminder_count` is already at the max.

4. **Root type: Missing field update in mutation.** The reassignment correctly updates identity fields (`approver_id`, `approver_name`, `assigned_at`) but omits all time-dependent fields.

## 3. Impact Assessment

- **Affected files/functions:**
  - `simple_api.py`: `reassign_approval_action()` (line 2131)
- **User-facing impact:** New approver sees an overdue or nearly-overdue action immediately after assignment. They may receive no reminders (if `reminder_count` is inherited at max). The urgency indicator in the UI is misleading.
- **Data impact:** No data corruption — the values are logically wrong but structurally valid. The fix is additive (updating fields, not restructuring).
- **Scope:** Every reassignment operation is affected. This is an admin-only operation (requires RFPO_ADMIN or GOD permission), so the frequency is relatively low, but the impact per occurrence is high because it breaks the SLA tracking for the affected approval.

## 4. Fix Strategy

### Phase 1: Core fix — Recalculate due_date on reassignment

After updating `assigned_at`, look up `timeout_days` from the instance_data snapshot and recalculate `due_date`:

```python
# Reassign
action.approver_id = new_approver.record_id
action.approver_name = new_approver.get_display_name()
action.assigned_at = datetime.utcnow()
action.updated_at = datetime.utcnow()

# Recalculate due_date from instance_data snapshot
timeout_days = 5  # default fallback
try:
    instance_data = action.instance.get_instance_data()
    for stage in instance_data.get("stages", []):
        if stage.get("stage_order") == action.stage_order:
            for step in stage.get("steps", []):
                if step.get("step_order") == action.step_order:
                    timeout_days = step.get("timeout_days", 5)
                    break
            break
except Exception as e:
    app.logger.warning("Could not resolve timeout_days for reassignment: %s", e)

action.due_date = action.assigned_at + timedelta(days=timeout_days)
```

### Phase 2: Hardening — Reset reminder and escalation state

Reset all time-dependent tracking fields so the new approver starts fresh:

```python
# Reset reminder/escalation state for fresh assignment
action.last_reminder_sent_utc = None
action.reminder_count = 0
action.is_escalated = False
action.escalated_at = None
action.escalation_reason = None
```

### Phase 3: Optional — Add reassignment audit detail

Include the old and new `due_date` in the audit trail so admins can see the deadline was recalculated:

```python
audit.set_details({
    ...
    "old_due_date": old_due_date.isoformat() if old_due_date else None,
    "new_due_date": action.due_date.isoformat(),
    "timeout_days": timeout_days,
})
```

## 7. Testing Strategy

### Regression tests
- **Test 1:** Reassign an action with `timeout_days=7` → `due_date` should be `now + 7 days`, not the original date
- **Test 2:** Reassign an action whose `due_date` has already expired → new `due_date` is in the future
- **Test 3:** Reassigned action has `reminder_count=0`, `is_escalated=False`, `last_reminder_sent_utc=None`
- **Test 4:** Reassign with corrupt/missing instance_data → `due_date` defaults to `now + 5 days`
- **Test 5:** Audit log contains `old_due_date`, `new_due_date`, `timeout_days`

### Manual verification
1. In the admin panel, reassign an action that is 3 days into a 5-day timeout
2. Verify the new approver's due_date shows 5 days from now (not 2 days)
3. Verify the action does NOT appear as overdue in the approver's dashboard

## 8. Regression Prevention

- Add assertion in reassignment test that `action.due_date > datetime.utcnow()` after any reassignment
- Consider adding a model-level method `reset_for_reassignment()` on `RFPOApprovalAction` that encapsulates all the fields that must be reset — prevents future field omissions
- If any new time-dependent fields are added to `RFPOApprovalAction` in the future, the reassignment function must be reviewed

## 9. Security Considerations

- **No security impact:** The fix only affects timing fields. Authorization checks remain unchanged.
- **Audit trail improved:** Adding due_date changes to the audit log gives admins better traceability for SLA compliance.
