## 2. Root Cause Analysis

The `withdraw_approval()` endpoint in `simple_api.py` (~line 2070-2130) uses `db.session.delete(instance)` to remove the `RFPOApprovalInstance` record when an RFPO is withdrawn from the approval process. The `RFPOApprovalInstance` model has a relationship to `RFPOApprovalAction` with `cascade="all, delete-orphan"` (defined in `models.py` line ~1952), so deleting the instance cascades to **permanently delete all approval actions** — including completed approvals, refusals, comments, conditions, and timing data.

**The defective code (simple_api.py ~line 2098):**
`python
# Delete the instance (cascades to actions)
db.session.delete(instance)
`

**Why this is wrong:**
1. **Destroys audit history** — Approval actions contain critical business records: who approved/refused, when, with what comments and conditions. These should never be deleted.
2. **The `AuditLog` entry for the withdrawal** only records that withdrawal happened, not the prior approval actions or their details.
3. **If the RFPO is resubmitted**, a brand new instance and actions are created with no link to the previous cycle. There's no way to answer "who approved this RFPO before it was withdrawn?"
4. **Compliance risk** — In regulated procurement environments, approval history must be retained for audit purposes regardless of withdrawal.

**The correct approach:** Instead of deleting the instance and its actions, the withdrawal should:
1. Set the instance's `overall_status` to `"withdrawn"` (soft-delete / status change)
2. Cancel any remaining pending actions by setting their status to `"cancelled"`
3. Preserve all completed actions (approved, refused, conditional) untouched
4. Reset the RFPO status to Draft as before

## 4. Fix Strategy

### Phase 1: Core Fix — Soft-delete via status change instead of hard-delete

Replace `db.session.delete(instance)` with a status-based withdrawal that preserves all records.

**Key changes to `withdraw_approval()` in `simple_api.py`:**

`python
# BEFORE (destructive):
db.session.delete(instance)

# AFTER (preserving audit trail):
# Mark the instance as withdrawn
instance.overall_status = "withdrawn"
instance.completed_at = datetime.utcnow()
instance.updated_at = datetime.utcnow()

# Cancel any pending actions (preserve completed ones)
for action in instance.actions:
    if action.status == "pending":
        action.status = "cancelled"
        action.completed_at = datetime.utcnow()
        action.updated_at = datetime.utcnow()
        action.comments = (action.comments or "") + "\n[System] Cancelled due to RFPO withdrawal." if action.comments else "[System] Cancelled due to RFPO withdrawal."
`

This preserves:
- All completed approval actions (approved, refused, conditional) with their comments and timestamps
- The instance record itself with its workflow configuration snapshot
- A clear "withdrawn" status for reporting and audit queries

### Phase 2: Hardening — Enhance the audit log entry

Snapshot the approval action summary into the `AuditLog` details so the withdrawal event itself contains a record of what was in progress:

`python
audit.set_details({
    "rfpo_id": rfpo.rfpo_id,
    "instance_id": instance.instance_id,
    "reason": request.get_json().get("reason", "") if request.is_json else "",
    "actions_at_withdrawal": [
        {
            "action_id": a.action_id,
            "step_name": a.step_name,
            "approver_name": a.approver_name,
            "status": a.status,
            "completed_at": a.completed_at.isoformat() if a.completed_at else None,
            "comments": a.comments,
        }
        for a in instance.actions
    ],
})
`

### Phase 3: Handle resubmission with withdrawn instances

When an RFPO is resubmitted for approval, the existing code checks for an existing instance. With the old delete approach, no instance exists. With the new approach, a "withdrawn" instance exists. The submission endpoint must handle this:

- Check if an instance exists with `overall_status == "withdrawn"`
- If so, create a **new** instance (with a new `instance_id`) rather than reusing the withdrawn one
- The RFPO's `approval_instance` backref uses `uselist=False`, so the old withdrawn instance's `rfpo_id` foreign key needs to be handled — either by nullifying it or by changing the backref to `uselist=True` to support multiple historical instances

**Recommended approach:** Change the relationship to allow multiple instances per RFPO, and add a helper method to get the "active" instance (the most recent non-withdrawn one). This is the cleanest long-term approach for audit trail preservation.

**Alternative minimal approach:** Before creating a new instance on resubmission, set the withdrawn instance's `rfpo_id` to NULL (orphan it from the RFPO but keep it in the DB for audit queries). This is simpler but makes it harder to query historical instances for a given RFPO.

**Recommended: Multiple instances approach**

In `models.py`, change the `RFPO` side of the relationship:
`python
# BEFORE (uselist=False — only one instance per RFPO):
rfpo = db.relationship(
    "RFPO", backref=db.backref(
        "approval_instance", uselist=False, cascade="all, delete-orphan"
    )
)

# AFTER (uselist=True — multiple instances, add helper):
rfpo = db.relationship(
    "RFPO", backref=db.backref(
        "approval_instances", lazy=True, cascade="all, delete-orphan"
    )
)
`

Then add a helper on RFPO:
`python
@property
def active_approval_instance(self):
    """Get the current (non-withdrawn) approval instance, if any."""
    if hasattr(self, 'approval_instances'):
        for inst in self.approval_instances:
            if inst.overall_status != "withdrawn":
                return inst
    return None
`

**IMPORTANT:** This Phase 3 change affects other parts of the codebase that reference `rfpo.approval_instance` (singular). All such references must be updated to use `rfpo.active_approval_instance`. A grep for `approval_instance` will identify all call sites.

## 5. Database Impact

### Phase 1 & 2: No DDL changes required
- The `overall_status` column already exists on `rfpo_approval_instances` and can hold `"withdrawn"`
- The `status` column on `rfpo_approval_actions` can already hold `"cancelled"`
- No ALTER TABLE needed

### Phase 3: No DDL changes required
- The relationship change (`uselist=False` → `uselist=True`) is a SQLAlchemy ORM-level change only
- The foreign key column `rfpo_approval_instances.rfpo_id` remains unchanged
- The database schema allows multiple rows with the same `rfpo_id` already (no UNIQUE constraint on `rfpo_id`)

**Data note:** Approval data destroyed by past withdrawals cannot be recovered. This fix only prevents future data loss.
