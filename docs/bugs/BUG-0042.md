## 1. Bug Details

| Field | Value |
|---|---|
| **Ticket** | BUG-0042 |
| **Title** | No idempotency protection on submit-for-approval |
| **Description** | The submit_for_approval() endpoint creates a new RFPOApprovalInstance without checking if one already exists for the RFPO. | **Steps to Reproduce** | 1. Submit an RFPO for approval. 2. The request times out on the client side (but server commits). 3. User retries the submission. 4. A second approval instance is created for the same RFPO. 5. Approvers receive duplicate email notifications. |

## 2. Root Cause Analysis

The `submit_for_approval()` endpoint at `simple_api.py:1728` **does have** an application-level check for existing instances (lines ~1766-1773):

```python
existing = RFPOApprovalInstance.query.filter_by(rfpo_id=rfpo.id).first()
if existing and existing.overall_status not in ("refused",):
    return jsonify({"success": False, "message": "This RFPO already has an active approval workflow"}), 400
```

However, this check has two critical weaknesses:

1. **Race condition (concurrent requests)**: Two simultaneous POST requests (e.g., user double-clicks, or client auto-retries after timeout) can both execute the `filter_by` query before either commits. Both see no existing instance, both proceed to create one. The `uselist=False` on the relationship backref is ORM-level only — it does **not** create a database-level unique constraint on `rfpo_approval_instances.rfpo_id`. SQLAlchemy will allow multiple rows with the same `rfpo_id`.

2. **Status check gap**: The RFPO status check (`rfpo.status not in ("Draft", "Refused")`) also reads without a lock. Between two concurrent requests, both read `status="Draft"`, both proceed, and both change it to "Pending Approval". The first commit succeeds; the second commit also succeeds because there's no conflict (both write the same value).

The practical scenario from the ticket: A client sends POST, the server processes it (PDF generation takes several seconds), commits, but the HTTP response times out. The client retries. The second request arrives AFTER the first committed, so the application-level check SHOULD catch it (RFPO status is now "Pending Approval"). **However**, the client-side retry may happen so fast that it arrives while the first request is still processing (before commit), creating the exact race condition described above.

Additionally, the `db.session.flush()` after creating the instance (line ~1896) makes the instance visible within the same transaction but NOT to other concurrent transactions (standard READ COMMITTED isolation). The `db.session.commit()` at line ~1992 finalizes everything, but during the ~50-100ms window between the flush and commit, a concurrent request reads no existing instance.

## 3. Impact Assessment

**Affected files/functions:**
- `simple_api.py:1728` — `submit_for_approval()` endpoint
- `models.py:1895` — `RFPOApprovalInstance` model (missing unique constraint)

**User-facing impact:**
- Approvers receive duplicate email notifications for the same RFPO
- Admin panel shows two approval instances for one RFPO — confusing workflow state
- If both instances proceed independently, the RFPO could be approved twice or have conflicting approval/refusal states
- The `uselist=False` backref means `rfpo.approval_instance` returns only ONE of the duplicates (whichever SQLAlchemy loads first), potentially hiding the duplicate

**Data impact:**
- Duplicate `RFPOApprovalInstance` rows for the same RFPO
- Duplicate `RFPOApprovalAction` rows with identical approver assignments
- Both instances send independent emails, creating confusion about which link to use
- Data integrity issue: two "active" instances for one RFPO violates the expected 1:1 relationship

**Scope:** Any RFPO submission under slow network conditions or heavy server load. Higher risk during PDF generation (which can take several seconds for complex RFPOs).

## 4. Fix Strategy

### Phase 1: Add database-level unique constraint (Core fix)
Add a partial unique index on `rfpo_approval_instances.rfpo_id` that only applies to non-refused instances. This is the definitive protection — even if the application-level check has a race condition, the database will reject the second insert.

```sql
-- Partial unique index: only one non-refused instance per RFPO
-- PostgreSQL syntax
CREATE UNIQUE INDEX uq_rfpo_active_instance
ON rfpo_approval_instances (rfpo_id)
WHERE overall_status NOT IN ('refused');
```

On the SQLAlchemy model, add the constraint:
```python
__table_args__ = (
    db.Index(
        'uq_rfpo_active_instance',
        'rfpo_id',
        unique=True,
        postgresql_where=db.text("overall_status NOT IN ('refused')"),
    ),
)
```

Then wrap the instance creation in a try/except to catch `IntegrityError` and return a 409 Conflict response with a meaningful message.

### Phase 2: Add SELECT FOR UPDATE lock (Hardening)
Before reading the RFPO and checking for existing instances, acquire a row-level lock on the RFPO row:

```python
rfpo = db.session.query(RFPO).filter_by(id=rfpo_id).with_for_update().first_or_404()
```

This ensures that only one transaction can read and modify the RFPO at a time. The second concurrent request will block until the first commits, then see the updated RFPO status ("Pending Approval") and existing instance.

**Note:** `SELECT FOR UPDATE` is a PostgreSQL feature. For local SQLite development, the partial unique index provides sufficient protection (SQLite serializes writes anyway).

### Phase 3: Client-side idempotency key (Optional improvement)
Accept an optional `Idempotency-Key` header from the client. Store it on the instance. If a duplicate key is received, return the existing instance instead of creating a new one.

```python
idempotency_key = request.headers.get('Idempotency-Key')
if idempotency_key:
    existing_by_key = RFPOApprovalInstance.query.filter_by(
        rfpo_id=rfpo.id, idempotency_key=idempotency_key
    ).first()
    if existing_by_key:
        return jsonify({"success": True, "instance": existing_by_key.to_dict()}), 200
```

This is the gold standard for idempotent APIs but requires client-side changes. Listed as optional.

## 7. Testing Strategy

**Regression tests:**
- Test that submitting an RFPO for approval succeeds and creates exactly one instance
- Test that re-submitting a "Pending Approval" RFPO returns 400/409
- Test that submitting after refusal creates a new instance (old one deleted)

**Edge case tests:**
- **Concurrent submission**: Two threads submit the same RFPO simultaneously → only one succeeds, the other gets 409 Conflict
- **Retry after timeout**: Submit succeeds server-side, client retries → second attempt returns 400 ("already has an active approval workflow") or 409
- **Submit after withdrawal**: RFPO withdrawn (status reset to Draft, instance deleted) → new submission succeeds
- **Submit refused RFPO**: Previous instance has `overall_status="refused"` → old instance deleted, new one created

**Manual verification:**
1. Deploy the partial unique index to production
2. Use `curl` to send two rapid-fire POST requests to `/api/rfpos/<id>/submit-for-approval`
3. Verify only one instance is created, second request gets 400 or 409
4. Check `rfpo_approval_instances` table — exactly one row for the RFPO

## 8. Regression Prevention

- The partial unique index is the permanent safeguard — it cannot be bypassed by any application code path
- Add integration test `test_submit_idempotent_on_retry` that explicitly tests the race condition scenario
- Document the idempotency behavior in API documentation
- Add a comment in `submit_for_approval()` explaining why both the application check AND the unique constraint are needed

## 9. Security Considerations

- **No SQL injection risk**: All queries use SQLAlchemy ORM with parameterized queries
- **No authorization bypass**: The `RFPO_ADMIN`/`GOD` permission check runs before any instance logic
- **Denial of service**: Without the fix, a malicious actor could create many duplicate instances by rapid-fire submissions, potentially overwhelming approvers with emails. The fix prevents this.
- **IntegrityError handling**: The catch block must NOT expose internal database details in the 409 response. Return only a user-friendly message.
