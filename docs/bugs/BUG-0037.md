## 2. Root Cause Analysis

The `reassign_approval_action()` endpoint in `simple_api.py` (~line 2140-2230) calls `send_approval_notification()` **synchronously** inside the request handler after `db.session.commit()`. The call is wrapped in a try/except (with a misleading `# Notify new approver (non-blocking)` comment), but it is NOT executed in a background thread.

**The defective code pattern (simple_api.py ~line 2209-2221):**
`python
# Notify new approver (non-blocking)  <-- MISLEADING COMMENT
try:
    from email_service import send_approval_notification
    rfpo = action.instance.rfpo
    if new_approver.email and rfpo:
        send_approval_notification(
            new_approver.email, new_approver.get_display_name(),
            rfpo.rfpo_id, f"Reassigned: {action.step_name}",
            rfpo_db_id=rfpo.id,
            context={...},
        )
except Exception as email_err:
    app.logger.warning(f"Email notification failed: {email_err}")
`

**Why this is wrong:**
- `send_approval_notification()` makes external HTTP calls to Azure Communication Services or SMTP servers, which can take 2-10+ seconds.
- The User App's `make_api_request()` helper has a hard 10-second timeout.
- If the email send takes longer than the remaining request budget, the response never reaches the caller, resulting in a "service unavailable" error.
- This is a **blocking call on the request thread**, not a background operation despite the comment claiming otherwise.

**Correct pattern (used by all other approval endpoints, e.g., line ~1510-1530):**
`python
# Collect email tasks as plain values (safe outside request context)
email_tasks = [(email, name, rfpo_id_str, subject, db_id, ctx_dict)]

# Build and return response FIRST
response_data = {"success": True, ...}

# Fire emails in background thread
if email_tasks:
    def _send_emails(tasks):
        from email_service import send_approval_notification
        for email, name, rfpo_id_str, subj, db_id, ctx in tasks:
            try:
                send_approval_notification(...)
            except Exception as e:
                logging.getLogger(__name__).warning(...)

    threading.Thread(target=_send_emails, args=(email_tasks,), daemon=True).start()

return jsonify(response_data)
`

This is a **consistency bug** — the pattern was not applied when the reassignment endpoint was written.

## 4. Fix Strategy

### Phase 1: Core Fix — Move email to background thread (the only required change)

Replace the synchronous email call with the standard background-thread pattern already used by all other approval endpoints in the same file.

**Key changes:**
1. Before `db.session.commit()`, snapshot all values needed for the email into a plain-value tuple (no ORM objects).
2. After commit, build the response dict.
3. Fire the email in a daemon background thread using the snapshotted values.
4. Return the response immediately.

**Code pattern:**
`python
# Snapshot values BEFORE returning (safe for background thread)
_email = new_approver.email
_name = new_approver.get_display_name()
_rfpo_id_str = action.instance.rfpo.rfpo_id if action.instance.rfpo else None
_step_name = action.step_name
_rfpo_db_id = action.instance.rfpo.id if action.instance.rfpo else None
_ctx = {
    'rfpo_id': action.instance.rfpo.id if action.instance.rfpo else None,
    'project_id': action.instance.rfpo.project_id if action.instance.rfpo else None,
    'consortium_id': action.instance.rfpo.consortium_id if action.instance.rfpo else None,
    'team_id': action.instance.rfpo.team_id if action.instance.rfpo else None,
}

db.session.commit()

# Build response
response_data = {
    "success": True,
    "message": f"Action reassigned from {old_approver_name} to {new_approver.get_display_name()}",
    "action": {
        "action_id": action.action_id,
        "approver_id": action.approver_id,
        "approver_name": action.approver_name,
    },
}

# Fire email in background thread
if _email and _rfpo_id_str:
    email_tasks = [(_email, _name, _rfpo_id_str, f"Reassigned: {_step_name}", _rfpo_db_id, _ctx)]

    def _send_emails(tasks):
        from email_service import send_approval_notification
        for email, name, rfpo_id_str, subj, db_id, ctx in tasks:
            try:
                send_approval_notification(email, name, rfpo_id_str, subj, rfpo_db_id=db_id, context=ctx)
            except Exception as e:
                logging.getLogger(__name__).warning("Background email to %s failed: %s", email, e)

    threading.Thread(target=_send_emails, args=(email_tasks,), daemon=True).start()

return jsonify(response_data)
`

### Phase 2: Hardening — Fix the misleading comment

Remove the `# Notify new approver (non-blocking)` comment that falsely claims the call is non-blocking, and replace with a comment explaining the background thread pattern.

## 10. Risk Assessment

| Risk | Severity | Mitigation |
|---|---|---|
| Background thread fails silently | Low | The existing `try/except` with logging in the `_send_emails` closure handles failures. Same pattern used by all other endpoints without issues. |
| ORM objects accessed in background thread | Medium | Mitigated by snapshotting all values into plain Python types (strings, ints, dicts) before starting the thread. No ORM objects cross the thread boundary. |
| Email sent after reassignment is rolled back | Very Low | Not possible — `db.session.commit()` is called before the thread starts. If commit fails, the except block rolls back and returns an error, never reaching the email code. |
